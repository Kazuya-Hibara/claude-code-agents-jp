## 1. 戦略的基盤

### 1.1 NEXUS が解決すること

個々のエージェントは強力です。しかし連携がなければ、次のような問題が発生します。
- アーキテクチャ判断の衝突
- 部門横断での作業重複
- ハンドオフ境界での品質ギャップ
- 共有コンテキストや組織的記憶の欠如

**NEXUS はこれらの失敗モードを排除します**。そのために以下を定義します。
- 各フェーズで **誰が** 起動するか
- **何を** 生成し、それが誰のためのものか
- **いつ** ハンドオフし、誰に渡すか
- 前進する前に品質を **どのように** 検証するか
- 各エージェントがパイプラインに存在する **理由**（不要な参加者をなくす）

### 1.2 コア原則

| 原則 | 説明 |
|-----------|-------------|
| **Pipeline Integrity** | 品質ゲートを通過しない限り、どのフェーズも前進しない |
| **Context Continuity** | すべてのハンドオフは完全なコンテキストを引き継ぐ。エージェントがゼロから開始することはない |
| **Parallel Execution** | 独立したワークストリームを並行実行し、タイムラインを圧縮する |
| **Evidence Over Claims** | すべての品質評価には主張ではなく証拠を必要とする |
| **Fail Fast, Fix Fast** | エスカレーション前のリトライはタスクごとに最大 3 回 |
| **Single Source of Truth** | 正規仕様は 1 つ、タスクリストは 1 つ、アーキテクチャドキュメントは 1 つ |

| **Legal Compliance Checker** | 規制スキャン | Compliance Requirements Matrix |
| **Tool Evaluator** | 技術ランドスケープ | Tech Stack Assessment |

```
WORKSTREAM A: Market Intelligence          WORKSTREAM B: User Intelligence
├── Trend Researcher                       ├── Feedback Synthesizer

    ├── Data handling constraints               ├── Build vs. buy analysis

### 3.3 Phase 0 品質ゲート

**Gate Keeper**: Executive Summary Generator

| 基準 | 閾値 | 必要な証拠 |

| 技術的実現可能性が確認済み | スタックが検証済み | Tool Evaluator assessment |

**アウトプット**: Executive Summary（500 語以内、SCQA 形式）→ 判定: GO / NO-GO / PIVOT

## 4. Phase 1 — 戦略 & アーキテクチャ

> **目的**: 1 行のコードを書く前に、何を構築するのか、それがどのように構成されるのか、成功が何を意味するのかを定義する。

### 4.1 アクティブなエージェント

| エージェント | フェーズ内の役割 | 主なアウトプット |
|-------|--------------|----------------|
| **Studio Producer** | 戦略的ポートフォリオ整合 | Strategic Portfolio Plan |
| **Senior Project Manager** | 仕様からタスクへの変換 | Comprehensive Task List |
| **Sprint Prioritizer** | 機能優先順位付け | Prioritized Backlog（RICE スコア付き） |
| **UX Architect** | 技術アーキテクチャ + UX 基盤 | Architecture Spec + CSS Design System |
| **Brand Guardian** | ブランドアイデンティティシステム | Brand Foundation Document |
| **Backend Architect** | システムアーキテクチャ | System Architecture Specification |
| **AI Engineer** | AI/ML アーキテクチャ（該当する場合） | ML System Design |
| **Finance Tracker** | 予算とリソース計画 | ROI 予測を含む Financial Plan |

### 4.2 実行シーケンス

```
STEP 1: Strategic Framing (Parallel)
├── Studio Producer → Strategic Portfolio Plan (vision, objectives, ROI targets)
├── Brand Guardian → Brand Foundation (purpose, values, visual identity system)
└── Finance Tracker → Budget Framework (resource allocation, cost projections)

STEP 2: Technical Architecture (Parallel, after Step 1)
├── UX Architect → CSS Design System + Layout Framework + UX Structure
├── Backend Architect → System Architecture (services, databases, APIs)
├── AI Engineer → ML Architecture (models, pipelines, inference strategy)
└── Senior Project Manager → Task List (spec → tasks, exact requirements)

STEP 3: Prioritization (Sequential, after Step 2)
└── Sprint Prioritizer → RICE-scored backlog with sprint assignments
    ├── Input: Task List + Architecture Spec + Budget Framework
    ├── Output: Prioritized sprint plan with dependency map
    └── Validation: Studio Producer confirms strategic alignment
```

### 4.3 Phase 1 品質ゲート

**Gate Keeper**: Studio Producer + Reality Checker（デュアルサインオフ）

| 基準 | 閾値 | 必要な証拠 |

## 7. Phase 4 — 品質 & ハードニング

> **目的**: 最終品質の難関。Reality Checker はデフォルトで「NEEDS WORK」と判断する。本番 readiness は圧倒的な証拠で証明しなければならない。

### 7.1 アクティブなエージェント

| エージェント | フェーズ内の役割 | 主なアウトプット |
|-------|--------------|----------------|
| **Reality Checker** | 最終統合テスト（デフォルトは NEEDS WORK） | Reality-Based Integration Report |
| **Evidence Collector** | 包括的な視覚的証拠 | Screenshot Evidence Package |
| **Performance Benchmarker** | 負荷テスト + 最適化 | Performance Certification |
| **API Tester** | 完全な API 回帰テストスイート | API Test Report |
| **Test Results Analyzer** | 品質メトリクスの集約 | Quality Metrics Dashboard |
| **Legal Compliance Checker** | 最終コンプライアンス監査 | Compliance Certification |
| **Infrastructure Maintainer** | 本番 readiness チェック | Infrastructure Readiness Report |
| **Workflow Optimizer** | プロセス効率レビュー | Optimization Recommendations |

### 7.2 ハードニングシーケンス

```
STEP 1: Evidence Collection (Parallel)
├── Evidence Collector → Full screenshot suite (desktop, tablet, mobile)
├── API Tester → Complete endpoint regression
├── Performance Benchmarker → Load test at 10x expected traffic
└── Legal Compliance Checker → Final regulatory audit

STEP 2: Analysis (Parallel, after Step 1)
├── Test Results Analyzer → Aggregate all test data into quality dashboard
├── Workflow Optimizer → Identify remaining process inefficiencies
└── Infrastructure Maintainer → Production environment validation

STEP 3: Final Judgment (Sequential, after Step 2)
└── Reality Checker → Integration Report
    ├── Cross-validates ALL previous QA findings

**判定オプション**:

## 12. 品質ゲート

### 12.1 ゲート概要

| フェーズ | ゲート名 | Gate Keeper | 通過基準 |
|-------|-----------|-------------|---------------|
| 0 → 1 | Discovery Gate | Executive Summary Generator | 市場が検証済み、ユーザーニーズが確認済み、規制上の道筋が明確 |
| 1 → 2 | Architecture Gate | Studio Producer + Reality Checker | アーキテクチャが完全、ブランドが定義済み、予算が承認済み、スプリント計画が現実的 |
| 2 → 3 | Foundation Gate | DevOps Automator + Evidence Collector | CI/CD が動作、スケルトンアプリが稼働、モニタリングが有効 |
| 3 → 4 | Feature Gate | Agents Orchestrator | すべてのタスクが QA を通過、重大バグなし、パフォーマンスベースライン達成 |
| 4 → 5 | Production Gate | Reality Checker（唯一の権限者） | ユーザージャーニーが完了、クロスデバイス一貫性あり、セキュリティ検証済み、仕様準拠 |
| 5 → 6 | Launch Gate | Studio Producer + Analytics Reporter | デプロイ成功、システム安定、グロースチャネルがアクティブ |

### 12.2 ゲート失敗時の処理

```
IF gate FAILS:
  ├── Gate Keeper produces specific failure report
  ├── Agents Orchestrator routes failures to responsible agents
  ├── Failed items enter Dev↔QA loop (Phase 3 mechanics)
  ├── Maximum 3 gate re-attempts before escalation to Studio Producer
  └── Studio Producer decides: fix, descope, or accept with risk
```

## Appendix C: NEXUS 用語集

| 用語 | 定義 |
|------|-----------|
| **NEXUS** | Network of EXperts, Unified in Strategy |
| **Quality Gate** | 証拠に基づく承認を必要とする、フェーズ間の必須チェックポイント |
| **Dev↔QA Loop** | 各タスクが前進前に QA を通過しなければならない継続的な開発・テストサイクル |
| **Handoff** | エージェント間で作業とコンテキストを構造化して引き渡すこと |
| **Gate Keeper** | フェーズ前進を承認または却下する権限を持つエージェント |
| **Escalation** | リトライを使い切った後、ブロックされたタスクを上位権限へルーティングすること |
| **NEXUS-Full** | すべてのエージェントを含む完全なパイプラインアクティベーション |
| **NEXUS-Sprint** | 機能/MVP 作業向けに 15-25 エージェントで行う集中的なパイプライン |
| **NEXUS-Micro** | 特定タスク向けに 5-10 エージェントを対象起動すること |
| **Pipeline Integrity** | 品質ゲートを通過しない限り、どのフェーズも前進しないという原則 |
| **Context Continuity** | すべてのハンドオフが完全なコンテキストを引き継ぐという原則 |
| **Evidence Over Claims** | 品質評価には主張ではなく証拠が必要であるという原則 |
